Marble Madness

Overveiw

During my work at Staffordshire University, I was tasked with understanding the fundamentals of Rapid games prototyping and used these ideas to design create and implement several different features/ game mechanics and demonstrate them in a working prototype.

Pachinko Peg

To Achieve this, I first created a list of mechanics I wanted to add to my game to enhance it as it was a framework based on a marble madness style game, I drew inspiration from other marble games. The first game mechanic was a pachinko peg that when hit by the user would bounce the user back, using lecture materials this was straight forward to implement, this led me to enhance the peg to not only bounce the player back but also be destroyed when this happened this allows the level to have breakable walls that feel substantial when the player collides with it.

Wiper Blade

The second feature I wanted to add was, what I dubbed, wiper blade. This wiper blade would be placed into the world space and move back and forth to ‘wipe’ the player off the map. This proved tricker to implement as I wanted to place it multiple times in various locations and currently, I was using public variables to achieve this, and it wasn’t the smooth movement I was looking for. This took me to the unity forums to research a potentially better way this led me to discover the Maths.PingPong function that is already bult into unity that would give me that steady and smooth back and forth that I wanted. I used this knowledge to create a second wiper that moved along the Z axis as well as the X allowing for greater customisability when it came to designing levels.

Magnet

Finally, I Wanted to create a magnet feature that would pull the player towards itself if the player got too close. This magnet would prove to be the most complex feature I wanted to add to my prototype. My initial implementation took me down a method of reversing the bounce peg system, using a vector3 to find out the direction and the reverse peg bounce to pull the user. Again, this gave me a uneven junky pull towards the magnet that would only trigger the initial tigger enter. Not for the first time during this project I thought there must be a better way, this led me back to the unity forums where I discovered the vector3.movetowards command. This made the whole process simpler, by using this command unity would smoothly pull the player towards the magnet whenever the trigger was true.
